## 4. 팀 구성 및 공정성 원칙

### 팀 구성 원칙

- 팀은 **교수자가 최종 배정**합니다.
- 사전 설문을 통해 다음 정보를 수집하여 가능한 한 반영합니다.
  - 전공, 코딩/수학 경험 (자기평가 1–5)
  - 함께 하고 싶은 사람 1–2명
  - 시간 제약 (야간 아르바이트, 통학 등)
- 특정 개인에 대한 "함께 하기 싫다" 식의 요청은 받지 않습니다.
- 각 팀에는 **최소 1명 이상의 코딩 경험자** 배치를 목표로 합니다.

### 배정 절차

1. **배정 기준**: 실력·전공·시간표를 섞어 팀 간 난이도 편차를 줄이는 방향으로 편성합니다.
2. 팀 배정 후, "구성 기준(예: 코딩 경험자 분산)"을 간단히 공지합니다.
3. **중도 팀 변경은 원칙적으로 불가**합니다. 다만 심각한 갈등·무임승차 등 특수 상황에서만 교수 판단 하에 조정할 수 있습니다 (성적 조정으로 대응하는 것을 원칙으로 함).

## 5. 프로젝트 요구사항

### 5-1. 문제 정의 및 데이터

- 각 팀은 **본전공과 관련된 지도학습 문제** (분류 또는 회귀)를 1개 정의합니다.
- 웹/공개 저장소에서 사용 가능한 데이터를 수집해야 합니다.
  - 개별 수집 데이터 사용 시 **사전 허가 필요** (아이디어 기안서 제출 시 함께 제출).
  - **최소 기준**: 변수(피처) 10개 이상, 인스턴스(행) 1,000개 이상 (전처리 이후 기준).
- 데이터 출처 (URL, 다운로드 날짜, 라이선스)를 명시해야 하며, **개인정보·민감 정보가 포함된 데이터는 사용 금지**입니다.

### 5-2. 모델링

- **기계학습 알고리즘 2개 이상**을 사용하여 비교합니다.
  - 예: 로지스틱 회귀, SVM, 랜덤포레스트, 그라디언트부스팅 등
- **선택 과제 (가산점)**: 딥러닝 모델 1개 추가 (TensorFlow/PyTorch + Colab, MLP/CNN/RNN 등 간단한 구조).
- 모든 모델은 **동일한 train/validation/test 분할**을 사용하고 **동일한 평가 지표**로 비교합니다.

### 5-3. 실험 및 분석

- SOTA 성능에 지나치게 집착하지 않아도 됩니다. 설명력 등 다양한 지표를 종합적으로 판단합니다.
- **성능 비교**: 문제에 적합한 지표 2개 이상 (예: 정확도, F1, ROC-AUC, (R)MSE 등).
- **시각화**: 최소 **8개 이상**의 그래프 포함 (EDA, 분포, 학습 곡선, 중요 변수, 오버피팅 분석 등).
- 과적합/언더피팅, 하이퍼파라미터 변경 효과, 모델 선택 이유를 보고서에서 설명합니다.

## 6. 발표 형식 및 제출 규정

### 발표 시간 구성 (팀당 15분 이내)

| 시간 | 내용 |
|------|------|
| 10분 | 프로젝트 개요, 데이터, 방법, 결과 발표 |
| 5분 | 질의응답 및 게임형 퀴즈 |

### 발표 일정

- 수업은 주 2회, 각 **75분** 수업.
- 한 수업에 **4팀 × 15분** 발표 진행.
- 총 8팀 → 한 주 내에 모두 발표 완료.
- 같은 시간에 발표하지 않는 **다른 4팀이 질문팀**이 됩니다.

### 게임형 퀴즈 진행 방식

질문팀은 각 발표팀을 하나씩 맡아 **객관식 또는 단답형 퀴즈 1문항**을 준비합니다.

> 질문 수준: "발표/제시한 코드 또는 알고리즘을 진지하게 들으면 맞힐 수 있을 정도"

#### 진행 순서

1. **발표 전 준비**: 질문팀은 맡은 발표팀의 주제·키워드를 확인하고 퀴즈 1문항을 준비합니다.
2. **발표 직후 Q&A**: 매칭된 질문팀이 퀴즈를 제시하고, 발표팀은 30초 내 상의 후 답합니다.
3. **점수 이동** (제로섬 구조):

| 결과 | 발표팀 | 질문팀 |
|------|--------|--------|
| 발표팀 **정답** | **+1점** | **-1점** |
| 발표팀 **오답/무응답** | **-1점** | **+1점** |

4. **점수 누적**: 한 수업에 발표 4개 → 각 팀 최대 **±4점** 출렁임.
   최종 성적 반영 시 **"게임형 퀴즈 10점 (±5점 한도)"** 안에서만 가감합니다.
